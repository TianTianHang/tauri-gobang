## Context

**当前状态：**
- `NetworkMessage` enum 定义了 10 种消息类型
- 其中 3 种（Chat, Join, GameOver）从未被实际使用
- 这些死代码在网络层代码中存在，但没有对应的前端实现或后端发送逻辑

**技术背景：**
- 联机模式使用 TCP + JSON 协议
- 消息通过 serde 序列化/反序列化
- 事件映射在 `handle_connection` 函数中
- 前端通过 Tauri events 接收网络消息

**约束：**
- 不影响现有的 7 种活跃消息类型
- 不破坏联机对弈功能
- 保持向后兼容（虽然这些消息未被使用）

## Goals / Non-Goals

**Goals:**
- 清理所有未使用的网络消息类型
- 减少协议定义的复杂度
- 提高代码可维护性

**Non-Goals:**
- 不修改任何活跃消息的实现
- 不改变网络协议的其他部分
- 不添加新功能

## Decisions

**决策 1: 完全删除而非注释**
- **选择**: 直接删除代码行
- **替代方案**: 注释掉并保留
- **理由**: 这些消息从未被使用，保留注释会增加认知负担；Git 历史已保留记录

**决策 2: 不添加迁移逻辑**
- **选择**: 直接删除，无兼容性处理
- **理由**: 这些消息从未被发送或接收，不存在兼容性问题

**决策 3: 删除所有相关代码**
- **选择**: 删除 enum 变体、事件映射、发送调用（如果存在）
- **理由**: 彻底清理，避免遗留引用

## Risks / Trade-offs

**风险 1**: 未来可能需要类似功能
- **缓解**: 如果未来需要，可以重新设计并实现，设计会更好（基于实际需求）

**风险 2**: 删除错误的代码
- **缓解**: 经过代码审查确认这些消息从未被使用：
  - Chat: 无发送命令，无前端监听
  - Join: 前端不监听 opponent_joined 事件
  - GameOver: 从未发送

**Trade-off**: 代码简洁度 vs 未来灵活性
- 接受略微增加的未来开发成本（重新实现）以换取当前的代码清晰度
